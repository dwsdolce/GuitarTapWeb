```
fix(status-bar): tap count restarted each phase; bar blinked; label froze at 0/N

Four bugs, one root: the status bar was ported from Swift's structure without its
bindings, so values were computed differently and refreshed at the wrong times.

0. is_detecting had a Swift-style didSet setter but NO signal, so the view was
   structurally unable to react to it — it could only sample it inside set_tap_count
   (driven by tapCountChanged). Swift gets this free (@Published re-renders the
   status bar). Add detectionStateChanged, emit it from the setter, bridge it through
   FftCanvas, and re-run set_tap_count on it. This is what makes 1 and 3 fixable.

1. The tap-count label ("0/3") froze for the whole sequence: its text was written
   only when `show` was true, and `show` samples is_detecting — which is False at
   every tap completion. Swift computes phaseLabel from current values and merely
   wraps it in an `if`, so its text can never be stale. Always write the text; gate
   only visibility.

2. current_tap_count = len(captured_taps) — but captured_taps is the WITHIN-phase
   buffer, cleared at every phase completion, so the count restarted at 0 on each
   phase change. Swift keeps the cumulative counter SEPARATE from the buffer
   (currentTapCount += 1). Consequences: the progress bar reset to 0 each phase, and
   the plate label's max(0, captured - (step-1)*number_of_taps) went negative and
   clamped — showing "Tap 0/N" from phase 2 onward. Fix: increment the counter, as
   Swift does. (The guitar path correctly keeps len(captured_taps) — that buffer is
   not cleared mid-sequence.)

3. The progress bar's visibility was gated on is_detecting, copied from Swift's *iOS*
   compactStatusBar. A desktop app must mirror fullStatusBar, which gates the bar on
   currentTapCount > 0 alone — so it appears on the first tap and stays up for the
   whole measurement. Fix: gate the bar on captured > 0.

4. The bar divided the cumulative count by number_of_taps instead of total_plate_taps,
   pinning it at 100% from the end of phase L. Fix: render analyzer.tap_progress.

5. The status bar used THREE different blues: the bar was palette(highlight) (light
   blue) while the labels beside it were a hardcoded rgb(40,100,210) (dark blue), and
   neither matched Swift. Swift draws the bar, the phase/tap label and the Peak
   readout as ONE colour (.blue). Pin Apple systemBlue via _system_blue(), a named
   token the Light/Dark/System theme work will own.

Add tests/test_tap_progress.py (3-way): the equations, the redo rebasing, and a
regression that drives the real gated-capture path across an Accept — verified to
FAIL on the old code and pass on the fix.
```
